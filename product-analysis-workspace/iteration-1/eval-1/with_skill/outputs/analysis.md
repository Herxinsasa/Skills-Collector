# 知识库自建产品分析

## 问题重定义

- **原始问题**：团队想自建一个内部知识库产品，统一管理文档、经验、会议纪要。
- **实际问题**：不是缺工具，是知识沉淀和检索的行为习惯没有建立起来。Notion 和 Confluence 是市面最成熟的知识管理产品，它们都没推起来，说明问题不在工具层。
- **关键矛盾**：团队把"工具不好用"当成了根因，实际根因极可能是"没人写、写了也没人看、看了也找不到"。换一个自研工具不会自动解决这三个问题。

> 核心追问：Notion/Confluence 推不起来的具体原因是什么？是上手门槛高？是搜索不好用？还是根本没人往里面写东西？如果是后者，自建工具只会重蹈覆辙，而且多花一套研发成本。

---

## 本质拆解

- **基本元素**：知识管理的本质只有三步——**记录**（有人把知识写下来）、**检索**（需要时能找到）、**保鲜**（过期内容被更新或标记）。这三步里，工具只影响"检索"的效率，"记录"和"保鲜"完全是人的行为问题。
- **类比陷阱**：
  - "我们需要一个 Wiki 一样的东西"——这是把 Confluence 的形态当成了本质。团队需要的不是 Wiki，是**信息能被找到**。
  - "自建可以完全贴合我们的工作流"——这是高估了定制化的价值，低估了研发和维护成本。贴合工作流不代表有人会用。
  - "Notion/Confluence 太重了"——这个判断成立的前提是，失败原因确实是"太重"。如果失败原因是"没人写"，那么再轻的工具也会失败。
- **本质推导方案**：如果抛开所有现有产品形态，从零出发——你要解决的是"让团队的知识能被记录和被找到"。这件事的最小方案不是做一个知识库系统，而是**把知识记录嵌入已有工作流**（比如会议纪要自动存档、重要群聊消息一键沉淀），以及**让检索覆盖所有已有数据源**（而不是要求大家迁移到一个新地方）。

---

## 剪枝分析

- **可复用（不自建）**：
  - 文档存储与协作：现有工具（飞书文档、Google Docs、Notion、钉钉文档等）已经足够好
  - 全文搜索：Elasticsearch / Meilisearch / Algolia 都是成熟方案
  - 权限管理：复用公司现有的 SSO/账号体系
- **可合并/精简**：
  - "文档 + 经验 + 会议纪要"不需要三个独立模块，统一为"条目 + 标签 + 全文搜索"即可
  - 编辑器和展示层可以合并，不存在本质差异
- **可推迟**：
  - AI 摘要/智能标签（初期手动打标签就够了）
  - 审批流程/版本管理（先用约定代替系统约束）
  - 知识图谱/关联推荐（需要大量内容才有价值）
- **可删除**：
  - 自研编辑器（用现有文档工具的嵌入或 Markdown 即可）
  - 自定义权限体系（初期全员可见就够，信息壁垒不是技术问题）
  - 移动端（内部知识库场景 95% 在桌面端）

### MVP 边界

- **必须有**：一个能跨数据源搜索的统一入口（对接已有文档系统 + 文件夹 + 聊天记录），以及一个极低摩擦的记录方式（比如 Slack/飞书 bot 一句话沉淀）。
- **绝对不要**：不要自建又一个"文档平台"，不要要求团队迁移现有文档，不要做任何需要"专门打开一个系统去写"的操作——摩擦每增加一步，使用率断崖下降。

---

## 综合结论

| 思维框架 | 核心洞察 |
|---------|---------|
| 苏格拉底（问问题） | 真正的问题不是"缺工具"，是"缺行为"。Notion/Confluence 已经证明成熟工具解决不了你们的行为问题，自建只会多亏一套研发成本。 |
| 第一性原理（拆问题） | 知识管理的本质是记录、检索、保鲜三件事，工具只在检索环节有边际贡献。核心瓶颈在"记录"和"保鲜"，这两件事靠流程和文化，不靠代码。 |
| 奥卡姆剃刀（做剪枝） | 不要做知识库产品。做一个极薄层：统一搜索入口 + 低摩擦记录方式。复用所有已有工具，不要求迁移，不引入新系统。 |

### 建议

- **定位**：不做"又一个文档平台"，做一个**跨数据源的搜索 + 一句话沉淀工具**。差异化在于零迁移成本和零额外操作习惯。
- **优先做**：先搞清楚 Notion/Confluence 失败的真实原因。用 5 个 Why 追问到根因，不要停在"不好用"这个层面。如果原因确凿是搜索/组织问题，才考虑技术手段；如果原因是"没人写"，应该先解决激励机制和流程习惯。
- **警惕**：最大风险不是做不出来，是做出来没人用——跟 Notion/Confluence 完全一样的结局，但这次额外付出了研发成本。**失败的代价不是 0，是有机会成本的。**
