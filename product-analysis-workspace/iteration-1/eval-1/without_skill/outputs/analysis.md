# 产品分析：自建内部知识库

## 待澄清的关键问题

在进入正式分析前，以下信息缺失会显著影响判断。分析中将对这些问题做合理推断，并标注假设。

1. **Notion/Confluence 推不起来的根因是什么？** 是工具太复杂？没人写？写了找不到？领导没推？还是根本不需要？
2. **"散落各处"具体指哪些地方？** 飞书文档、微信聊天、本地文件、Git 仓库、邮件、还是口头传递？
3. **团队规模和组成？** 10 人团队和 200 人团队的知识管理策略完全不同。
4. **自建动机是什么？** 觉得现有工具太贵、功能太多想简化、还是想深度集成内部系统？

> **核心假设（后续分析基于此）：** 现有工具失败的主因不是功能缺失，而是团队缺乏文档写作习惯和维护动力。团队规模在 30–100 人之间，以工程师为主。

---

## 阶段一：苏格拉底提问 — 把问题问对

### 追问一：你确定问题出在"工具"上吗？

Notion 和 Confluence 是目前全球最成熟的知识管理产品，分别服务了数千万用户和数十万企业。如果这两个产品在一个团队里都推不起来，最不可能的原因就是"工具不够好"。一个自研团队要追上其中任何一个产品的基础体验，保守估计需要 5–10 名工程师投入 6–12 个月。而即便追上，团队面对的还是同一群人、同一套工作习惯。

**这不是工具问题，这是行为改变问题。**

### 追问二：谁会写？谁会用？为什么？

知识库的三方博弈：

- **写的人**：需要花时间整理、结构化、维护更新。收益是间接的（别人能找到答案），且往往不归自己。
- **读的人**：需要能快速找到准确答案。如果找不到，就会直接问人，然后文档就更没价值了。
- **维护的人**：需要持续清理过时内容、整理分类体系。几乎没人愿意干这个。

Notion 和 Confluence 推不起来，大概率是这三方博弈中的某一环断裂了。最常见的断裂点是**没人写**：文档是额外负担而非工作流的一部分。

### 追问三：不做会怎样？

如果继续现在的状态——文档散落各处，靠口头传递和 IM 搜索——团队还能正常运转吗？如果能，说明知识管理对这个团队来说不是刚需，强推任何工具都会失败。如果已经明确出现了严重影响（新人入职学习周期过长、关键知识随离职流失、重复踩坑），那才值得投入。

### 问题重定义

| 维度 | 内容 |
|------|------|
| 原始问题 | 要不要自己做一个知识库产品？ |
| 实际问题 | **如何让团队产生可持续的知识沉淀和复用行为？** |
| 关键矛盾 | 知识管理是慢反馈的正外部性行为——写文档的成本由个人当下承担，收益由团队在未来获得。任何工具方案如果不能打破这个激励错配，结果都会和 Notion/Confluence 一样。 |

## 阶段二：第一性原理 — 回归本质

### 拆解问题本质

把"知识管理"拆到最底层，只有四个基本元素：

1. **捕获** — 信息如何进入系统（人写？自动采集？AI 生成？）
2. **组织** — 信息如何分类、关联、结构化
3. **检索** — 需要的时候如何找到
4. **保鲜** — 如何保证信息不过时

绝大多数自建知识库的错误在于：把 80% 的精力花在第 2 步（分类体系、目录结构、权限模型、Markdown 编辑器），而第 1 步和第 4 步几乎为零投入。结果是建了一个漂亮的空图书馆，没有人往里放书。

### 识别类比陷阱

| 类比陷阱 | 为什么是陷阱 |
|----------|-------------|
| "要有树形目录结构" | Confluence 的空间-页面层级继承自 Wiki 时代，但大多数人的心智模型是搜索而非浏览。 |
| "要有富文本/Markdown 编辑器" | 编辑体验不是瓶颈。飞书文档和 Notion 的编辑器已经做到了极致，但依然有很多团队写不起来。 |
| "要有权限体系" | 权限是发现的对立面。内部知识库的默认假设应该是全开放，权限控制应最小化。 |
| "要做成 Wiki" | Wiki 假设有人持续维护。现实中，知识库更像"日志流"而非"百科全书"。 |
| "要有标签系统" | 标签在没有强力治理的情况下一定会熵增，最终变得毫无意义。 |

### 从本质重构

如果抛开 Confluence、Notion 等产品的形态，从四个基本元素出发，一个有效的内部知识管理系统应该长这样：

- **捕获**：最低摩擦。理想状态下，知识生产应该发生在工作流内部而不是额外操作。例如：代码评审中的讨论自动归档、会议录音自动转写并提取要点、IM 中解决的问题自动沉淀为 FAQ。**人不应被要求"去写文档"，文档应该是工作的副产品。**
- **组织**：最小结构。不做前置分类，依靠搜索 + AI 摘要来降低对目录的依赖。允许混乱存在，让结构从使用中涌现而非从设计中强加。
- **检索**：极致准确。AI 语义搜索是刚需，关键词搜索是底线。如果搜不到，就等于不存在。
- **保鲜**：自动化。定期扫描文档最后更新时间，自动标记可能过时的内容。内容与代码、项目状态关联，当相关代码变更时自动提醒更新。

**核心洞察：知识库产品竞争的不是编辑器或目录树，而是"捕获成本"和"检索精度"。谁能让写文档这件事从"主动行为"变成"被动副产品"，谁就能解决留存问题。**

## 阶段三：奥卡姆剃刀 — 做剪枝

### 逐项剪枝

**可复用（不自建）：**

- 编辑器（富文本/Markdown）→ 用开源方案，不要自研
- 全文搜索 → Elasticsearch/Meilisearch 等成熟方案
- 文件存储 → 对象存储（S3/MinIO）
- AI 语义搜索 → embedding 模型 + 向量数据库（已有成熟组合）
- 权限与认证 → 对接公司已有的 SSO/账号体系，不要自己造

**可合并/精简：**

- 目录树 + 标签 + 搜索 → 主界面就是一个搜索框 + AI 对话，目录和标签作为辅助而非核心
- 多个"创建"入口（文档/页面/知识库/空间）→ 一个入口：写就是了，分类是后来的事
- 评论/协作/通知 → 对接已有的 IM（飞书/企业微信/Slack），不重复造通知系统

**可推迟：**

- 富文本协作编辑 → 第一版只要 Markdown + 版本记录就够了
- 复杂的权限体系 → 第一版全员可见即可
- 数据分析仪表盘 → 如果没有内容，仪表盘毫无意义
- 移动端 → 内部知识库 95% 的读写都在 PC 上完成
- 多语言/国际化 → 内部工具不需要

**可删除：**

- 自定义工作流引擎
- 模板市场
- 导入 Confluence/Notion 数据（迁移成本极高且一次性的功能，投入产出比极低）
- 点赞/收藏/积分/排行榜 → 在知识库场景，社交功能几乎总是噪音

### MVP 边界

| 类别 | 内容 |
|------|------|
| **必须有** | 极简的 Markdown 编写 + AI 语义搜索 + 与现有 IM 打通（IM 里搜得到知识库、IM 里写的内容能一键沉淀）+ 内容过时自动提醒 |
| **绝对不要** | 复杂的目录层级、精细的权限控制、社交互动功能、自定义工作流、移动端 App、数据迁移工具 |

## 综合结论

| 思维框架 | 核心洞察 |
|---------|---------|
| 苏格拉底（问问题） | 两次失败说明问题不在工具而在习惯。自建工具不会自动让人开始写文档——就像换一个牌子的健身器材不会让人开始健身。 |
| 第一性原理（拆问题） | 知识库的底层竞争要素是"捕获成本"和"检索精度"，不是"编辑体验"。让文档成为工作的副产品而非额外任务，是唯一的破局路径。 |
| 奥卡姆剃刀（做剪枝） | 如果一定要做，MVP 应该是一个搜索框 + 一个 Markdown 输入框 + AI 语义检索 + IM 集成。其他一切都可以砍掉或先用开源方案垫着。 |

### 最终建议

**不要自建。** 这是核心判断。

理由不是"你们做不出来"，而是"即使做出来一个技术优秀的产品，大概率会重蹈 Notion/Confluence 的覆辙"。知识库产品在技术实现上没有护城河，真正的壁垒在组织行为层面。把工程资源花在攻克一个组织行为问题而非技术问题上，是资源的错配。

**建议的替代路径（按优先级排列）：**

1. **诊断先行（投入：1–2 周）。** 找 5–10 个同事做深度访谈，搞清楚 Notion/Confluence 失败的根因。是不知道怎么用、觉得没用、没时间写、还是找不到想要的内容？不同根因对应完全不同的解法。
2. **如果根因是习惯/文化问题**，需要的不是新工具，而是一个"知识管理负责人"（兼职即可），以及和绩效/复盘流程绑定的文档机制。比如：每个项目结项必须有复盘文档、每次故障必须有 postmortem、新人入职 3 天内找不到答案的算知识库缺陷。用一种已有工具（哪怕是一个共享文件夹 + 全局搜索）把这个机制跑通 3 个月，再考虑要不要做工具。
3. **如果根因真的是工具不匹配**（比如核心痛点是"会议录音无法自动转文字沉淀"、"代码和文档无法关联"、"IM 里的讨论无法沉淀"），那么**不要做通用知识库，做一个解决单一痛点的微型工具**。比如一个 Slack/飞书 bot，有人问过的问题自动记录为 FAQ，下次有人问类似问题时自动推送。这个小工具跑通了，再逐步扩展。
4. **如果实在坚持要自建**，严格遵循 MVP 边界：一个搜索框 + 一个输入框 + AI 语义搜索 + IM bot 集成。先让 5 个人用起来，观察 1 个月，再决定是否继续投入。绝不在验证行为改变之前建城堡。

**最大的风险：** 投入 6 个月开发、功能齐全、体验不错，但依然没人用。届时损失的不是开发成本，而是团队对"知识管理"这件事的信心和耐心——第三次失败后，这个组织可能永远不再尝试做知识沉淀了。
