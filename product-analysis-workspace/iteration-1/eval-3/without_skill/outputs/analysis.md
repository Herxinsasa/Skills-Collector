# Scrum 转 Kanban 决策分析：该不该换、怎么换

---

## 一、问题诊断：你的 Scrum 到底哪里出了问题？

在回答"该不该换"之前，必须先问"为什么痛"。你描述了三个关键信号：

1. **节奏越来越累** -- 不是某一次冲刺累，而是持续半年的累积疲劳。
2. **回顾会形式大于实质** -- 这是最危险的信号。回顾会是 Scrum 的"纠错机制"，当它失效，Scrum 就失去了自我改进能力。
3. **跑了半年** -- 排除了"还在磨合期"的解释。团队已经过了学习曲线，问题不是不熟练，而是模式本身可能不适配。

### 可能的根因分析（按概率排序）

**根因 A：冲刺长度与工作类型不匹配（高概率）**

两周冲刺适合"可独立交付的增量开发"。但如果你的团队同时承担：
- 线上问题修复（打断冲刺计划）
- 跨团队依赖（等待外部接口）
- 需求频繁变更（利益相关方在冲刺中途改主意）

那么每两周就会上演一次"冲刺末赶工 + 不完整交付 + 回顾会抱怨同样的问题"的循环。冲刺边界从"保护团队"变成了"制造焦虑"。

**根因 B：仪式感侵蚀（中高概率）**

两周一个冲刺意味着每个月有两次完整的仪式循环：规划会 + 每日站会 x10 + 评审会 + 回顾会。当这些会议产出的价值低于时间成本时，团队自然会觉得"形式大于实质"。尤其是回顾会，如果每次提出的改进项在下个冲刺依然无人跟进，它就变成了集体吐槽大会。

**根因 C：估算军备竞赛（中概率）**

故事点估算、速率追踪、燃尽图……这些工具在复杂度和不确定性高的环境中，会产生"精确的错误"。团队花了大量时间争论一个 Backlog 条目是 3 点还是 5 点，但这些时间本可以用来写代码。

**根因 D：冲刺承诺的心理负担（需团队确认）**

"冲刺目标不可变"是 Scrum 的铁律，但现实世界不总是配合。如果团队反复陷入"承诺了但做不完 → 回顾会反省 → 下个冲刺继续做不完"的循环，说明固定的冲刺边界正在伤害团队士气，而不是保护它。

## 二、Scrum vs Kanban：核心差异对照

| 维度 | Scrum（两周冲刺） | Kanban（持续流动） |
|------|-------------------|---------------------|
| **节奏** | 固定时间盒，每两周一个闭环 | 无固定节奏，按需流动 |
| **计划** | 冲刺规划会确定冲刺待办列表 | 就绪队列 + 定期补货会议 |
| **承诺** | 冲刺目标承诺（不可变） | 无承诺，按能力拉取 |
| **限制方式** | 时间盒限制（冲刺结束时必须停止） | WIP 限制（同时进行中的条目上限） |
| **度量** | 速率（velocity）、燃尽图 | 前置时间（lead time）、周期时间（cycle time） |
| **角色** | SM、PO、开发团队（3 个角色） | 无强制角色 |
| **核心会议** | 5 个强制仪式（规划/站会/评审/回顾/梳理） | 最少 2 个（站会 + 补货会议） |
| **改进机制** | 每次冲刺结束的回顾会 | 持续改进（可随时调整流程） |
| **最佳场景** | 目标清晰、可预估的增量开发 | 需求多变、运维支持、持续交付 |

## 三、核心问题：Kanban 是不是换汤不换药？

**不是。但有个前提条件。**

Scrum 和 Kanban 虽然都是敏捷框架，但它们的"约束理论"立足点完全不同：

- Scrum 的约束是**时间**（timebox）-- 固定时间，弹性范围。冲刺结束时强制交付，做不完就溢出。
- Kanban 的约束是**在制品**（WIP）-- 固定 WIP，弹性时间。任何时候团队只专注于有限条目，完成一个才拉取下一个。

这意味着两种完全不同的压力模型：

- Scrum 压力来自"冲刺最后三天还没做完"，团队被迫加班或砍范围。
- Kanban 压力来自"WIP 满了没法拉新任务"，团队被迫去帮别人完成卡住的条目。

如果你的团队疲劳主要来自冲刺末期的 deadline 压力，Kanban 的 WIP 约束能从根本上改变这个动力模型。**这绝不是换汤不换药。**

但如果你的团队疲劳来自：
- 需求方不断插队（Kanban 也需要明确的优先级排序）
- 技术债导致的交付速度下降（换框架不能替代还债）
- 团队协作问题（换流程不能替代沟通）

那么换了 Kanban 之后，这些问题会换一种形式继续存在。

## 四、决策框架：该不该换？

### 建议切换的信号（满足 3 条以上建议认真考虑）

- [ ] 冲刺期间频繁被打断（线上问题、紧急需求）
- [ ] 冲刺结束时经常有未完成的条目，蔓延到下一冲刺
- [ ] 回顾会上反复提出相同的问题但没有改进
- [ ] 估算花费大量时间但预测价值很低
- [ ] 团队的业务本质是"持续服务"而非"阶段性项目"
- [ ] 站会变成了汇报进度的过场，而非发现障碍
- [ ] 冲刺规划会上的需求在冲刺中途就过时了

### 建议暂留 Scrum 的信号

- [ ] 团队自组织能力弱，需要冲刺节奏来维持纪律
- [ ] 利益相关方强烈依赖冲刺评审会这一固定节点
- [ ] 项目有硬性里程碑日期，冲刺是向上汇报的基础单元
- [ ] 团队刚稳定下来，再变一次流程可能造成更多混乱

### 折中方案：Scrumban

如果你倾向于"先试试水温"，Scrumban 是一个低风险的过渡选择：

- **保留**：站会、回顾会（调整为月度或按需）
- **去掉**：冲刺规划会 + 冲刺承诺（改为持续从就绪队列拉取）
- **新增**：WIP 限制（从小开始，比如"每人同时最多 2 个任务"）
- **保留但弱化**：冲刺评审会（改为按需的 demo 日）

这样，团队先体验到 WIP 限制带来的减压效果，再决定是否要完全去掉冲刺边界。

## 五、实施路径：怎么换？

如果你决定切换，以下是分阶段实施路径：

### 第一阶段：准备（1-2 周）

1. **数据基线采集**：在切换之前，记录当前状态的目标数据。
   - 平均前置时间（从需求提出到上线）
   - 平均周期时间（从开发开始到上线）
   - 每个冲刺的吞吐量（实际完成的条目数）
   - 冲刺末期加班频率
   没有这些基线，你无法量化切换的效果。

2. **可视化当前流程**：画出现有的工作流（待办 → 开发 → 测试 → 上线），标记出瓶颈环节。这是你设计 Kanban 列的基础。

3. **团队共识**：开一次专门的讨论会（不是回顾会），坦诚讨论：
   - 当前 Scrum 哪些部分让大家疲惫？
   - 对 Kanban 的了解和期望是什么？
   - 最担心切换后发生什么？
   
   **必须让团队参与决策，而不是"管理层决定换了"。**

### 第二阶段：试点（4-6 周）

1. **建立物理或数字看板**：列设置建议如下。
   - 待办（Backlog）
   - 就绪（Ready for Dev）-- 已澄清、就绪的条目，团队从此列拉取
   - 开发中（In Progress）-- **设置 WIP 限制**
   - 测试中（In Testing）-- **设置 WIP 限制**
   - 完成（Done）

2. **设定初始 WIP 限制**：让团队自己决定，原则是"比现状稍微紧一点但不至于喘不过气"。一个常见的起点：
   - `开发中` WIP = 团队成员数 × 0.8（比如 5 人团队设 4）
   - `测试中` WIP = 团队成员数 × 0.5（测试通常是瓶颈，先设紧一点）

3. **保留最少的仪式**：
   - 每日站会：仍然 15 分钟，但在 Kanban 板前开（"走板"而非"走人"）
   - 每周补货会议：30 分钟，补充"就绪"列
   - 每两周回顾会：保留一段时间，观察节奏变化

4. **停止估算**：用"条目数量"替代"故事点"，关注吞吐量（每周完成多少条目）而非速率（每冲刺完成多少故事点）。这是切换 Kanban 后最大的心理和操作转变。

5. **建立度量体系**：
   - 累积流图（CFD）-- 一个图同时展示 WIP、前置时间、瓶颈
   - 前置时间分布图 -- 看 85%/95% 的条目在多长时间内完成
   - 吞吐量追踪 -- 稳定后可作为交付承诺的依据

### 第三阶段：调优（持续进行）

1. **根据数据调整 WIP 限制**：如果"测试中"列持续堆积，说明 WIP 设太高或测试是瓶颈，需要针对性地解决（而非单纯限流）。
2. **逐步废除不必要的仪式**：如果回顾会仍然形式化，试着改成"看数据说话"——每次只回顾 CFD 图的变化趋势，15 分钟内结束。
3. **定义服务等级（Classes of Service）**：对于紧急需求，设定"快速通道"（但严格限制同时数量），避免所有需求都标成紧急。

## 六、常见陷阱与对策

| 陷阱 | 表现 | 对策 |
|------|------|------|
| **WIP 限制形同虚设** | 领导说"这个紧急，破例加一个"，最终 WIP 翻倍 | WIP 突破必须记录原因，每月回顾"破例率"，超过阈值触发流程调整 |
| **没有回顾的 Kanban 变成放任** | 去掉了回顾会后，流程问题无人关注 | 用数据替代主观感受：累积流图出现"沙漏形"就触发流程讨论 |
| **站会变长** | 在 Kanban 板前讨论变成深入技术讨论 | 主持人严格限制：只说"堵了没、需要谁帮"，细节会后单独沟通 |
| **切换后短期吞吐量下降** | 团队在适应新流程 | 预期内。前 2-4 周的吞吐量不应作为判断依据 |
| **利益相关方不适应** | 没了冲刺评审会，需求方不知道什么时候能看到交付 | 改为"按需 Demo"或"每周开放日"，保证透明度 |
| **以为切了 Kanban 就不用改善了** | 换完流程就结束了，不再迭代 | Kanban 的本质是"持续改进"，没有回顾会不等于没有回顾 |

## 七、终极建议

**回到第一性原理：你的目的是什么？**

不是"要不要用 Kanban"，而是"如何让团队的交付节奏可持续，且保持持续改进的能力"。

基于你给出的信息（半年 Scrum、疲劳累积、回顾会形式化），我的判断是：

**该换。但不是因为 Scrum 不好，而是你的团队当前阶段适合更轻量的流程。**

理由：
1. 回顾会形式化说明 Scrum 的"检视与适应"循环已经断裂。这是 Scrum 最核心的引擎，引擎坏了，整个框架就只剩空壳。
2. 半年是一个足够长的时间窗来判断"磨合问题"还是"不适配问题"。如果前 3 个月累是磨合，后 3 个月还累就是模式问题。
3. 团队已经通过 Scrum 建立了基本的敏捷心智（任务拆分、可视化、站会沟通），这些在切换到 Kanban 后不会丢失。换句话说，Scrum 已经完成了它的"训练作用"。

### 具体行动建议

**立刻这周可以做的：**
- 在下次回顾会上明确提出切换讨论，让团队每个人说"我支持/反对/有条件的支持"，并说明理由。
- 如果团队支持，就从**下一个冲刺结束后**开始试点 Kanban，而不是中途切换。

**第一个月的验收标准：**
- 团队成员主观感受：压力是否下降？（最关键的指标，比任何数据都重要）
- 前置时间是否趋于稳定？（不一定要下降，但不能剧烈波动）
- 是否有"被遗忘的条目"？（Kanban 没有冲刺边界，需要额外关注长期滞留的条目）

**三个月后的"回头看"：**
- 如果切换后团队疲劳感明显下降、交付节奏稳定 → 说明决策对了，继续深化。
- 如果疲劳感仍在但变成了另一种形式 → 可能是团队或组织层面的深层问题（如需求质量、技术债、人员配比），需要跳出"Scrum vs Kanban"的二元框架来解决。
- 如果团队觉得失去冲刺边界后反而迷失方向 → 立刻切回 Scrum，或引入 Scrumban 补回一些节奏感。**回退不丢人，用错工具才丢人。**

## 八、总结

| 问题 | 回答 |
|------|------|
| 该不该换？ | **该换**，你的症状指向 Scrum 的核心循环已失效 |
| 是不是换汤不换药？ | **不是**，Scrum 约束时间、Kanban 约束 WIP，压力模型本质不同 |
| 怎么换？ | 数据基线 → 试点 → 调优，三阶段渐进切换，不要一刀切 |
| 最大风险？ | WIP 限制形同虚设 + 没了回顾会之后流程不再改进。两者都需要用数据驱动的方式防范 |
| 什么时候判断切换失败了？ | 三个月后团队依然累、或失去方向感。届时果断切回或引入 Scrumban |
